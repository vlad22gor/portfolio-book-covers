# Project Rules

## Общие правила

- `tasks/projrules.md` фиксирует устойчивые правила сайта, а не историю экспериментов.
- Для изменений правил и архитектурных договорённостей обязательна краткая запись в `tasks/logs.md`.
- Постоянные архитектурные договорённости храним в `docs/`, а рабочие заметки и исследования — в `tasks/`.

## Сайт

- Сайт остаётся `static-first` проектом на `Astro + TypeScript + Tailwind`.
- Источник контента кейсов: `src/content/cases/*.md`.
- `Motion` является первичным инструментом для любого runtime-motion, UI transitions и stateful animation на сайте.
- `video/webm/mp4` допустимы как медиа-ассет или заранее отрендеренный art-directed motion, но не как замена UI/state animation.
- Клиентский JS и islands добавляем только там, где без них нельзя реализовать motion, interaction или runtime state; статические блоки и контент остаются в `.astro`.
- На контентных страницах сохраняем приоритет минимального runtime JS; `Motion` используем осознанно, а не по привычке.
- `Dialkit` разрешён только локально и в dev-режиме как tuning-инструмент; он не должен попадать в production UX.
- Любые значения, найденные через `Dialkit`, считаются черновыми, пока не перенесены в shared production constants/config.

## Motion-компоненты

- Для motion-компонентов используем 3-слойную схему: `View`-слой для общего presentation/UI, `Dev`-слой для `Dialkit` и tuning controls, `Prod`-слой для той же `View` с зафиксированными production values.
- Источник правды для найденных motion-параметров живёт отдельно от `Dialkit`-конфига: в shared config/constants файле.
- Нельзя дублировать JSX, вёрстку и presentation-логику между dev/prod версиями компонента; отличается только источник параметров и wiring.
- Новые интерактивные компоненты не считаются готовыми без dev-сценария, в котором их можно проверить и докрутить.

## Dev-каталог компонентов

- В dev обязателен отдельный каталог компонентов по умолчанию на маршруте `/dev/components`.
- Каталог компонентов существует только в dev/local preview и не публикуется в production.
- В каталоге должны быть все ключевые компоненты, их варианты, состояния, интерактивные сценарии и отдельные tuning-scenes под `Dialkit`.
- Каталог проектируем как рабочее место для доводки, а не как маркетинговый styleguide: важны состояния, edge cases и motion-переходы.

## Доступность

- `icon-only` кнопки обязаны иметь `aria-label`.
- Form controls обязаны иметь `<label>` или `aria-label`.
- Интерактивные элементы обязаны быть клавиатурно доступны.
- Для действий используем `<button>`, для навигации — `<a>`/`Link`, а не `div onClick`.
- Изображения обязаны иметь `alt`; декоративные изображения могут иметь пустой `alt`.
- Декоративные иконки обязаны иметь `aria-hidden="true"`.
- Асинхронные обновления должны использовать `aria-live="polite"` или другой подходящий live-region.
- Любой motion и transition обязан иметь reduced-motion поведение через `prefers-reduced-motion`.
- Визуально эффектный motion не должен ломать доступность, keyboard flow и reduced-motion сценарии.

## Проверки

- Изменения сайта проверяем через `npm run astro -- check` и `npm run build` из корня репозитория.
- Если документация в `docs/` расходится с этими правилами, сначала ориентируемся на `tasks/projrules.md`, затем синхронизируем `docs/`.
